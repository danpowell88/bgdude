bgdude — Engineering process

How work flows from an idea to verified-done: the GitHub Issues pipeline, the agent loops, the merge gates, and the human verification batches. Source of truth for conventions is CLAUDE.md; standing choices live in doc/decisions/ (this page reflects decisions 6–13, updated 2026-07-11). Tracking lives in GitHub Issues + the bgdude project board — statuses are status:* labels, Done = the issue closed by Summer. The pre-migration Backlog.md IDs map to issues via doc/backlog-migration-map.md.

1. Status pipeline decision-12

Ideaanyone→ Plannedgroomer→ To Dogroomer→ Doingimplementer→ Needs Reviewimplementer→ Reviewingreviewer→ Reviewedreviewer (merged!)→ Requires Human Verificationgroomer batches→ DoneSummer only
  • Idea — a raw thought. No ACs required; just capture it (gh issue create --label "status:idea").
  • Planned — groomed: description, testable ACs, plan, milestone, ordinal, dependencies. Issues needing answers wait here with the detail-needed label.
  • To Do — ready: dependencies met, spec sufficient. The only status implementers pick from, lowest ordinal first.
  • Doing — claimed by one implementer: flip the label to status:doing + post a signed claiming comment. Means actively worked right now.
  • Needs Review — implementation finished: verify pipeline green, branch pushed, PR open.
  • Reviewing — claimed by one reviewer (same label-flip + signed comment — no two reviewers race an issue).
  • Reviewed — review passed and the PR is merged to main. The agent pipeline is complete; awaiting batch verification. A failed review goes back to To Do instead (branch + PR stay open, findings attached).
  • Requires Human Verification — grouped into a verification-batch issue for Summer (see §6).
  • Done — Summer verified the batch on the real app and closes the issues. Agents never close a task issue (PRs say Refs #n, never Closes #n, so a merge can't auto-close past her gate).
  • Blocked — orthogonal parked state from any active stage; the comment must name the blocker and what would unblock it.

Merging happens at Reviewed, not after human verification — main stays reviewed and CI-green at all times, while the human gate verifies outcomes in batches without serializing merges.

3. A task's life, step by step

Implementer (one task per run)

  1. Pull main; pick the lowest-ordinal To Do issue with deps met.
  2. Claim first: flip to status:doing + post a signed claiming comment naming branch issue-<n> — before any code; back off if an earlier claim is already there.
  3. Work in an own worktree on branch issue-<n>.
  4. Implement per plan/ACs; log friction: comments; keep doc/user-guide.html current.
  5. Merge origin/main in, run the full verify pipeline (§5).
  6. Push branch, open PR issue-<n>: <title> (Refs #n), record PR: #<pr>, tick ACs, closing comment ending with a friction: line, flip to status:needs-review.

Reviewer (drains the queue)

  1. Skip anything it implemented itself (decision-7).
  2. Claim: flip to status:reviewing + signed comment.
  3. Freshen a stale/conflicting branch (git merge origin/main, push, let CI re-run).
  4. Review the PR diff adversarially against ACs/DoD (sweep-the-whole-surface checklist); notes on the PR and the issue.
  5. Pass: reviewed-by: comment, DoD ticked, gh pr merge --merge --delete-branch (never --admin), flip to status:reviewed — not closed.
  6. Fail: reviewed-by: comment with concrete problems, no merge, back to To Do — the next implementer resumes on the same branch/PR.

Groomer (front + back of pipeline)

  1. Flesh out Idea issues into Planned (desc, ACs, plan, milestone, ordinal, deps); split anything bigger than one PR.
  2. Promote Planned → To Do when deps are met and the spec is implementable.
  3. Batch Reviewed issues (≥5, a completed milestone, or >7 days old) into a verification-batch issue for Summer; members → Requires Human Verification.
  4. Apply Summer's verdicts: pass → close members + batch (Done); problems → file fix issues, bounce affected members to To Do.

Summer (human gate)

  1. A batch issue assigned to you appears in Requires Human Verification.
  2. Install the latest debug APK (artifact on the newest main CI run) or flutter run.
  3. Walk the batch checklist — one checkbox per member issue saying exactly what to verify.
  4. All good → close members + batch (or tell an agent to). Problems → comment what's wrong on the batch; the groomer files the fixes.

4. Git, PRs & the merge gate decisions 6, 8, 10, 11

  • All work in worktrees + branches. Every writer works in its own git worktree on its own branch off the latest main — one branch per issue, named issue-<n> (pre-migration branches use the old task-<id> Backlog names — see doc/backlog-migration-map.md).
  • Every task ships as a GitHub PR. Implementers never push code to main, never merge; the reviewer loop merges via gh pr merge --merge only.
  • The GitHub ruleset "main merge gate: PR + green CI" machine-enforces it. Merging to main requires:
    • a pull request (merge-commit method only; no force-push / deletion of main);
    • required checks green: analyze (with --fatal-infos), coverage-gate, apk-build, native-tests, actionlint (workflow lint), android-lint (Android Lint, warningsAsErrors + committed baseline), dependency-review (blocks known-vulnerable dependency changes) — decision-14;
    • CodeQL code-scanning results present and free of medium+ security / warning-level alerts introduced by the PR (decision-14 tightened from high+/errors) — the security-and-quality suite means quality findings block too (the free stand-in for GitHub's Code Quality rule, which needs an org on Team/Enterprise);
    • coverage on the PR at/above the 65% floor and not below the latest successful main run — the no-drop ratchet (the free stand-in for GitHub's "Restrict code coverage" rule).
  • Straight-to-main is only for rare bookkeeping — doc/decisions/ entries, config edits, trivial docs. The admin bypass on the ruleset exists solely for these (pushes print "Bypassed rule violations" — expected). Using it to merge a PR (--admin) or push task code is forbidden.
  • Label flips are the coordination protocol (decision-13): the instant an issue changes status, its status:* label is updated and a signed comment posted — an atomic API call every parallel agent sees instantly. (This replaced the old claim-commit churn on main, which was making feature branches go stale and conflict — old TASK-310.)

5. The verify pipeline (local must match CI)

  1. flutter pub get
  2. dart run build_runner build --delete-conflicting-outputs (generated files are not committed)
  3. flutter analyze --fatal-infos — zero diagnostics of any severity (strict-casts / strict-raw-types / strict-inference are on)
  4. flutter test --coverage test/ — green, and coverage at/above the floor and not below main (CI enforces the ratchet on PRs)
  5. flutter build apk --debug
  6. Kotlin changed? cd android && ./gradlew :app:testDebugUnitTest and :app:lintDebug (new warnings fail — the committed baseline only covers pre-existing ones)

CI additionally runs CodeQL (security + quality) on every PR and push to main. Integration tests (integration_test/) run nightly on an emulator and locally — they are not a merge gate (decision-5).

6. Human verification batches (Summer) decision-12

Agent review confirms the code is correct and merged; the human gate confirms the product is actually what was intended. To keep this cheap it is batch-level, not per-ticket:

  • The groomer groups related Reviewed tasks into one batch task — title Human verification batch <date>, label verification-batch, assigned to Summer — whose acceptance criteria are a checklist: one line per member task with exactly what to check on the real app.
  • Batches are coherent (same feature/milestone where possible) so you verify a piece of work, not a grab-bag.
  • Triggers: ≥5 Reviewed tasks, a milestone fully Reviewed, or anything sitting Reviewed for a week.
  • Your verdict is recorded on the batch task (check the ACs / leave a comment). Pass → everything moves to Done. Problems → the groomer turns each into a prioritised fix task and bounces the affected member back to To Do.

7. Starting agents (scripts & schedules)

One command: scripts/run-agent.ps1

pwsh scripts/run-agent.ps1                                # one implementer run (claude, sonnet)
pwsh scripts/run-agent.ps1 -Role reviewer                 # one reviewer run (claude, opus)
pwsh scripts/run-agent.ps1 -Role groomer                  # groom ideas + build verification batches
pwsh scripts/run-agent.ps1 -Agent qwen                    # implementer via qwen code instead of claude
pwsh scripts/run-agent.ps1 -Loop -IntervalMinutes 45      # keep implementing, one task per iteration
pwsh scripts/run-agent.ps1 -Role reviewer -Model opus -Loop -IntervalMinutes 60

Each iteration launches a fresh agent that reads CLAUDE.md, follows loops/<role>.md end-to-end (pick → claim → work → hand off), and exits. Because every claim is an instant label flip + signed comment on the issue, you can run several instances — even across machines, or a mix of Claude and qwen — without them stepping on each other.

Fully unattended: Windows Task Scheduler

schtasks /Create /TN "bgdude-implementer" /SC HOURLY /TR "pwsh -NoProfile -File C:\Users\danpo\Desktop\bgdude\scripts\run-agent.ps1 -Role implementer" /F
schtasks /Create /TN "bgdude-reviewer"    /SC HOURLY /ST 00:30 /TR "pwsh -NoProfile -File C:\Users\danpo\Desktop\bgdude\scripts\run-agent.ps1 -Role reviewer" /F
schtasks /Create /TN "bgdude-groomer"     /SC DAILY  /ST 07:00 /TR "pwsh -NoProfile -File C:\Users\danpo\Desktop\bgdude\scripts\run-agent.ps1 -Role groomer" /F

Watching without interfering

Everything is live on GitHub: the issues list (filter by label:status:doing etc.), the main bgdude backlog board (project #2 — Status columns for every open issue, sortable by Ordinal), and one board per milestone (projects #12–#20, titled after the phases, including their Done issues so per-phase progress reads at a glance). No local mirror needed — watching never touches a working tree.

8. Conventions quick reference

ConventionRule
Claim protocolStatus change → flip the status:* label + signed comment, immediately, before any other work; re-read and back off if someone claimed first. A pushed branch whose issue is still To Do is a bug.
Identity tagsimplemented-by: / reviewed-by: comments + a leading <agent-id>: on every comment + git Co-Authored-By, always the agent's own id (e.g. Sonnet, Fable, qwen-code).
Friction logfriction:<build|env|deps|code|test|tooling> — what bit you — root cause/fix comment as you hit it; the closing comment before Needs Review must end with one (or friction:none).
detail-neededDon't guess. Label + concrete questions; the issue waits in Planned (or Blocked if it was Doing).
Bug fixesSweep the whole surface: sibling call sites, both branches/side-effects, a test that fails if reverted, concurrency/security nuance. Reviewers re-apply the same four axes.
OrdinalsBand 1 (100000+) stabilise/fix/test → band 2 (500000+) finish existing features → band 3 (700000+) net-new. Pick lowest ordinal with deps met.
User guideAny user-visible change updates doc/user-guide.html in the same PR.
DecisionsStanding choices live in doc/decisions/ — read before re-litigating; record new ones as a numbered markdown file (straight-to-main lane).
