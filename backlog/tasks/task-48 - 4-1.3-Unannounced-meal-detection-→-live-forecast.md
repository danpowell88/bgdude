## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**Background.** If you eat without logging it, your glucose rises for a reason bgdude can't see, so the forecast misses it and the learning code mistakes the unlogged meal for insulin resistance. There's already a "meal detector" in the code, but nothing uses it and it has no tests.

**Reason for change.** Spotting an unannounced meal lets the forecast bend to what's actually happening and stops the app mislabelling unlogged food as resistance — all without ever adding phantom carbs to the dose calculator or your records.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->

- [ ] #2 Synthetic carbs bend the forecast cone only, never advisor/persistence

- [ ] #4 Autotune no longer reads unlogged meals as resistance

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
**Technical notes.** (a) Run MealDetector over the last ~45 min each refresh; on detection append a synthetic CarbEntry INSIDE the forecast-state builder ONLY (never reaches advisor COB or persistence); Predict shows a chip ("assuming ~35 g unlogged — confirm?"). (b) Stage a PendingConfirmation; confirming writes the real entry via quick-log, dismissing KV-suppresses that window. (c) Pass detected windows into Autotune.analyseDay as pseudo-carbs. Safety: ≥15 min sustained, cap ~60 g, decay when the rise stops.

**Testing.** MealDetector unit tests (first); bent-cone + staged-confirmation provider test; Autotune-no-longer-inflated test. Assert synthetic carbs never reach the advisor or DB. ML-honesty tests (coverage + bias, synthetic recovery); guard the forecaster promotion gate; `flutter analyze`/`flutter test` green.
