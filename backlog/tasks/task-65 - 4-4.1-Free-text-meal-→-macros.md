## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**Background.** Logging a meal means looking up each item's carbs. There's no way to just type "chicken burrito and chips" and get an estimate.

**Reason for change.** Free-text-to-macros, using the same on-device AI as the label scanner, removes the biggest friction in logging meals. It reuses the scanner's safety checks so estimates stay grounded, is always editable, and falls back to a food-name search when no AI model is installed.
<!-- SECTION:DESCRIPTION:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
**Technical notes.** New buildMealEstimatePrompt (dish list → itemised JSON) on the Gemma runtime, greedy decoding, 45 s timeout, close-after-use; shares §5's validation/grounding (items 1–3 prerequisite). Entry: a text field on Meals/advisor, always editable, labelled an estimate; no-model fallback = AFCD name search. Degrades, never blocks.

**Testing.** Prompt→JSON parse test; validation bounds applied; AFCD fallback when no model; the estimate is always editable. Validation/grounding tests (bounds + OCR-grounding); degrade gracefully with no model; `flutter analyze`/`flutter test` green.
