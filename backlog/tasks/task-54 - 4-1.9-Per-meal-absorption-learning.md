## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
**Background.** Different meals digest at different speeds (a salad vs a pizza), which changes how long carbs keep raising your glucose. bgdude assumes one fixed absorption time, even though it already records how each saved meal actually played out.

**Reason for change.** Learning each meal's real absorption time — and applying it when you log that meal again — improves the forecast, carb-on-board and learning for that meal, essentially for free.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [ ] #1 absorptionMinutes fit from ≥3 outcomes
- [ ] #2 Stamped on CarbEntry when logging that meal
- [ ] #3 Meal sheet shows learned absorption
- [ ] #4 Fit-recovery + round-trip tests
<!-- AC:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
- Fit `absorptionMinutes` ≈ clamp(2·timeToBaseline − preBolus, 90, 360), median of ≥3 outcomes, stored on the `SavedMeal`.
- Logging that meal stamps `CarbEntry.absorptionMinutes`.
- Meal sheet shows "your absorption ≈ 4.5 h (6 meals)"; the FPU coach skips its extension when learned duration already exceeds it.
- Tests: fit-recovery test; persistence round-trip; differing-COB-tail test; ML-honesty tests (coverage + bias, synthetic recovery); guard the forecaster promotion gate.
- Verify: `flutter analyze` clean, `flutter test` green.
<!-- SECTION:PLAN:END -->
